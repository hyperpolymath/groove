== How conformance works

Conformance is defined as an *executable checklist*. Every requirement below
carries a stable ID (`CONF-<level>-<nn>`); the reference test suite implements
exactly one test per ID (`provider/tests/conformance.rs`, function names
`conf_l1_01` … `conf_l2_07`), run against the reference provider
(`groove-provider`). CI enforces that every ID in this document has a matching
test.

[source,bash]
----
cargo test --workspace        # runs the conformance suite (and everything else)
----

An implementation claims a level by passing every MUST at that level and all
levels below it, exercised against a live instance.

== Level 1: Groove-Aware (discovery)

[cols="1,4",options="header"]
|===
| ID | Requirement
| CONF-L1-01 | `GET /.well-known/groove` MUST return `200` with a Groove Manifest; the default `Content-Type` is `application/groove+json` (SPEC §2.1.2).
| CONF-L1-02 | The JSON manifest MUST carry `groove_version` = `"1"`, `service_id`, `service_version`, `mode`, and a `capabilities` object (SPEC §2.1.1–2.1.2).
| CONF-L1-03 | The manifest MUST offer at least one capability whose `type` is a registered capability type (`registry/groove-registry.json`).
| CONF-L1-04 | Content negotiation: `Accept: application/groove+a2ml` MUST return the A2ML rendering if the provider offers it; an absent or unknown `Accept` MUST still yield the JSON manifest (bare-by-default, SPEC §2.1.3).
|===

A Level 1 system additionally MUST function without any groove partners
(bare-first) — exercised implicitly: the provider under test starts alone.

== Level 2: Groove-Connected (lifecycle)

[cols="1,4",options="header"]
|===
| ID | Requirement
| CONF-L2-01 | `POST /.well-known/groove/connect` with a structurally compatible consumer manifest (consumer `consumes` ⊆ provider offers) MUST return `200` and a JSON body containing a connection handle id (SPEC §4.2).
| CONF-L2-02 | An incompatible connect (consumer consumes a capability type the provider does not offer) MUST return `409 Conflict` with machine-readable reasons, and MUST NOT mint a handle.
| CONF-L2-03 | `GET /.well-known/groove/heartbeat` MUST return `204 No Content` within 500ms (SPEC §4.3).
| CONF-L2-04 | `POST /.well-known/groove/disconnect` with a valid handle MUST return `200` and linearly consume the handle: a second disconnect with the same handle MUST return `410 Gone` (SPEC §4.5).
| CONF-L2-05 | Graceful degradation: after a disconnect, `GET /.well-known/groove` MUST still return `200` — losing a partner is not an error (SPEC §4.4).
| CONF-L2-06 | Dual-stack: the manifest endpoint MUST answer on both `[::1]` and `127.0.0.1`; clients MUST probe `[::1]` first (TRANSPORT §7.6).
| CONF-L2-07 | Attestation: connect and disconnect MUST each emit a provenance record carrying `event`, `provider`, `consumer`, `timestamp`, `hash`, `prev_hash`; consecutive records MUST hash-chain — record _n_'s `prev_hash` equals record _n−1_'s `hash` (SPEC §5.1).
|===

A Level 2 system additionally MUST emit `groove:connected` /
`groove:disconnected` events internally (SPEC §2.4) and SHOULD implement
local port-probe discovery.

== Aspirational levels (not yet testable)

The former Levels 3 and 4 are preserved here as targets. They CANNOT be
normative while no implementation — including the reference one — meets them;
no conformance IDs are assigned until an executable test exists for each
line item.

=== Level 3: Groove-Proven (target)

. Capability interfaces defined in Idris2 with dependent types
  (blocked on PROOFS-1, see `proofs/README.adoc`)
. Linear types for connection handles (the reference provider approximates
  linearity at runtime per CONF-L2-04)
. C headers generated from the Idris2 ABI; Zig FFI layer for C-compatible
  bindings
. No `believe_me` / `assert_total` in groove ABI code

=== Level 4: Groove-Native (target)

. Version negotiation with SemVer constraints
. Capability-based security requirements
. Distributed discovery (mDNS/DNS-SD or BEAM clustering)
. Transitive mesh composition with type-safety proofs
. Provenance stored in VeriSimDB (when available via groove)

== Self-Test

Every groove-aware system SHOULD include a self-test command that verifies
its manifest, exercises a connect/disconnect cycle against itself, verifies
graceful degradation, and reports the conformance level achieved. The
reference implementation's equivalent is the conformance suite itself.
