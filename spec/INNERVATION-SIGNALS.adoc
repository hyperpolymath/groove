== Abstract

Innervation signals are the vocabulary of the hyperpolymath nervous system.
Every repo-level event that crosses the boundary between producer and
consumer (extractor → VeriSimDB, CI → Hypatia, ECHIDNA → proof store) travels
as a Groove message with a name defined in this schema.

This document lists the v0.1 canonical signal names, their payloads, and
their producers and consumers.

== Naming Convention

Signal names use dot-separated lowercase segments:

----
<domain>.<entity>.<event>
----

- `<domain>` names the subsystem (`crg`, `proof`, `compliance`, `ecosystem`,
  `annotation`, `commit`, `ci`, `echidna`, `hypatia`)
- `<entity>` names the thing acted upon (`grade`, `verified`, `finding`,
  `link`, `extracted`, `pushed`)
- `<event>` names the transition (`new`, `changed`, `resolved`, `stale`)

Two-segment names are allowed when the event is obvious from the entity
(e.g. `commit.pushed`, `ci.completed`).

== Message Envelope

All signals travel as Groove messages with this envelope (independent of
the body schema above):

----
@groove_message(version="1.0"):
signal: <canonical-signal-name>
producer: <service-name>
timestamp: <ISO-8601-UTC>
correlation_id: <UUID>         # optional; joins related messages
body: { ... signal payload ... }
@end
----

The `correlation_id` is used to thread a chain of signals (e.g.
`commit.pushed` → `ci.completed` → `compliance.finding.new` all share an id).

== Versioning

- Schema version is carried in this document (v0.1.0) and in each message
  envelope.
- Adding a new signal is a minor-version bump (v0.1.0 → v0.2.0).
- Removing or renaming a signal is a major-version bump.
- Adding an optional field to a payload is a patch bump.

== Consumer Registration

A service consuming a signal registers interest via its
`coordination.k9` `innervation.efferent` section:

----
innervation:
  efferent:
    - signal: "echidna.proof.verified"
      action: "ingest certificate into proof-status octad"
      auto_fix: false
----

The Groove registry uses these declarations to route messages.

== Unknown Signals

Consumers MUST ignore unknown signal names. Producers SHOULD NOT emit
signals not listed here without first updating this document.
