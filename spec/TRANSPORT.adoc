== Overview

This document specifies the transport-layer requirements for Groove Protocol
discovery, connection, and heartbeat traffic. Groove operates over HTTP on
the loopback interface for local discovery, and optionally over mDNS/DNS-SD
for LAN discovery. Correct handling of IPv4 and IPv6 dual-stack is critical
for reliable discovery across all operating systems and browser environments.

== 1. Binding Requirements

=== 1.1. Dual-Stack Binding (REQUIRED)

Groove-aware services MUST bind to dual-stack sockets that accept both
IPv4 and IPv6 connections.

**Preferred:** Bind to `[::]:PORT` with dual-stack enabled (the operating
system default on Linux, macOS, and modern Windows). This creates a single
socket that accepts connections from both `127.0.0.1` (IPv4) and `::1`
(IPv6).

**Alternative:** If the operating system does not support dual-stack sockets,
the service MUST bind to BOTH `127.0.0.1:PORT` and `[::1]:PORT` explicitly.

**Prohibited:** A Groove service MUST NOT bind exclusively to one protocol
family. A service that binds only to `127.0.0.1` is invisible to IPv6
probes. A service that binds only to `[::1]` is invisible to IPv4 probes.

[source,zig]
----
// Correct: dual-stack binding in Zig
const addr = std.net.Address.parseIp6("::", port) catch unreachable;
var server = std.net.StreamServer.init(.{ .reuse_address = true });
server.listen(addr) catch |err| { /* handle */ };
----

[source,elixir]
----
# Correct: dual-stack binding in Elixir/Phoenix
config :my_app, MyAppWeb.Endpoint,
  http: [ip: {0, 0, 0, 0, 0, 0, 0, 0}, port: 6473]  # [::] in tuple form
----

=== 1.2. Port Allocation

Groove services SHOULD use ports in the range 6460–6500 for local discovery.
This range is currently unassigned by IANA.

Well-known port assignments within the Groove ecosystem:

[cols="1,2,3"]
|===
| Port | Service | Protocol

| 6470 | Gossamer | Groove + WebView IPC
| 6473 | Burble | Groove + WebRTC + BEAM
| 6475 | VeriSimDB | Groove + REST + gRPC
| 6480 | Vext | Groove + Attestation
|===

Services outside the Groove ecosystem SHOULD use the range 8080–8100 as a
secondary probe range.

== 2. Discovery Probing

=== 2.1. Probe Order (IPv6-Preferred)

When probing for Groove partners, the discovery engine MUST follow this order:

. Probe `[::1]:PORT` (IPv6 loopback) with a 500ms timeout
. If no response, probe `127.0.0.1:PORT` (IPv4 loopback) with a 500ms timeout
. If either responds with a valid Groove Manifest, the partner is discovered

**Rationale:** Modern operating systems (Fedora 43, macOS Sonoma, Ubuntu 24.04,
Windows 11) resolve `localhost` to `::1` first via Happy Eyeballs (RFC 8305).
Probing IPv6 first aligns with OS behaviour and avoids unnecessary fallback
latency on the majority of systems.

=== 2.2. Browser Extension Probing

Browser extensions present unique constraints:

. The `fetch()` API resolves `localhost` according to the browser's own
  resolver, which may differ from the OS resolver
. **Firefox:** Tries `::1` first, falls back to `127.0.0.1` (Happy Eyeballs)
. **Chrome:** Behaviour varies by OS network stack; may prefer IPv4 on some
  configurations
. **Safari:** Restricted by App Transport Security; may block non-HTTPS
  localhost on some configurations

To ensure reliable discovery from browser extensions:

. Extensions SHOULD probe `http://[::1]:PORT` and `http://127.0.0.1:PORT`
  explicitly, rather than relying on `http://localhost:PORT` resolution
. Extensions MUST handle `fetch()` failures gracefully (the port is not
  groove-aware, not an error condition)
. Extensions SHOULD use `AbortController` with a 500ms timeout per probe

[source,javascript]
----
// Correct: explicit dual-stack probing from a browser extension
async function probePort(port) {
  // Try IPv6 first
  var manifest = await probeSingle('[::1]', port);
  if (manifest) return manifest;
  // Fallback to IPv4
  return await probeSingle('127.0.0.1', port);
}

async function probeSingle(host, port) {
  try {
    var controller = new AbortController();
    var timer = setTimeout(function() { controller.abort(); }, 500);
    var resp = await fetch('http://' + (host.includes(':') ? '[' + host + ']' : host)
      + ':' + port + '/.well-known/groove', {
      signal: controller.signal,
      headers: { 'Accept': 'application/groove+a2ml' }
    });
    clearTimeout(timer);
    if (resp.ok) return await resp.text();
  } catch (e) { /* not groove-aware, skip */ }
  return null;
}
----

== 3. CORS Requirements

=== 3.1. Mandatory CORS Headers

All Groove HTTP endpoints (`/.well-known/groove/*`) MUST include CORS
headers to allow browser extensions and web applications to probe them.

[source,http]
----
HTTP/1.1 200 OK
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type, Accept
Access-Control-Max-Age: 86400
Content-Type: application/groove+a2ml
----

**Rationale:** Without CORS headers, browser extensions cannot `fetch()`
from the extension context to localhost services. The `*` origin is safe
here because these endpoints are on the loopback interface — only software
on the same machine can reach them.

=== 3.2. Preflight Handling

Groove services MUST respond to `OPTIONS` preflight requests on all
`/.well-known/groove/*` paths with the CORS headers above and a
`204 No Content` status.

=== 3.3. LAN CORS

For LAN discovery (services on other machines), the CORS `Origin` header
SHOULD be restricted to known groove origins rather than `*`. This prevents
arbitrary websites from probing LAN groove services.

[source,http]
----
# LAN groove endpoint — restricted origin
Access-Control-Allow-Origin: moz-extension://{extension-id}
----

== 4. LAN Discovery (Optional)

=== 4.1. mDNS/DNS-SD

For discovery beyond localhost, Groove services MAY register a DNS-SD
service record:

[source]
----
_groove._tcp.local. IN SRV 0 0 6473 my-service.local.
_groove._tcp.local. IN TXT "id=burble" "version=1.2.0" "caps=voice,text,presence"
----

**IPv6 multicast:** mDNS queries are sent to `ff02::fb` (IPv6) or
`224.0.0.251` (IPv4). IPv6 multicast is RECOMMENDED as the primary
transport for LAN discovery.

**Rationale:** Every device on a LAN has an IPv6 link-local address
(`fe80::...`) automatically — no DHCP required. This makes IPv6
mDNS the most reliable zero-configuration LAN discovery mechanism.

=== 4.2. BEAM Clustering

Elixir/Erlang systems MAY use native BEAM node discovery for distributed
Groove. BEAM clustering provides:

. Automatic node discovery via `epmd` (Erlang Port Mapper Daemon)
. Built-in message passing between nodes
. No additional discovery protocol needed

When BEAM clustering is used, Groove manifests are exchanged via
`GenServer.call/2` rather than HTTP.

=== 4.3. Security Boundary

LAN discovery MUST be explicitly opted into. The default configuration
MUST be localhost-only. This prevents unintended exposure of groove
capabilities to other machines on the network.

[source,a2ml]
----
@groove-manifest(version="0.1.0"):
  @system(id="my-service", name="My Service", version="0.1.0")
  @discovery:
    @scope("localhost")           # Default: loopback only
    # @scope("lan")              # Opt-in: LAN discovery via mDNS
    # @scope("wan")              # Future: WAN discovery (not specified)
  @end
@end
----

== 5. Connection Transport

=== 5.1. HTTP for Control Plane

All Groove control-plane traffic (discovery, connect, disconnect, heartbeat)
uses HTTP/1.1 over TCP.

HTTPS is NOT required for localhost connections (the loopback interface is
inherently trusted). HTTPS SHOULD be used for LAN connections and MUST be
used for WAN connections (if specified in future).

=== 5.2. Custom Transport for Data Plane

Once a Groove connection is established, the data plane (actual capability
traffic) MAY use any transport:

. HTTP/REST — for simple request/response
. WebSocket — for bidirectional streaming
. gRPC — for high-performance RPC
. WebRTC — for voice/video (Burble)
. Unix domain sockets — for same-machine high-throughput
. BEAM message passing — for Elixir systems

The transport for the data plane is declared in the capability interface
definition (Idris2 ABI). The Groove control plane only handles discovery
and lifecycle — it does not mandate or proxy data-plane traffic.

== 6. IPv6 Transition Considerations

=== 6.1. IPv6-Only Systems

On systems with no IPv4 stack (increasingly common in cloud and container
environments), Groove services MUST function correctly using only IPv6
loopback (`::1`). The discovery engine MUST NOT fail if IPv4 probes return
network-unreachable errors.

=== 6.2. IPv4-Only Systems

On legacy systems with no IPv6 stack, Groove services MUST function
correctly using only IPv4 loopback (`127.0.0.1`). The discovery engine
MUST NOT fail if IPv6 probes return network-unreachable errors.

=== 6.3. Dual-Stack Systems (Default)

On dual-stack systems (the common case), the binding and probing rules
in Sections 1 and 2 apply. IPv6 is preferred, IPv4 is the fallback.

== 7. IPv6 Transition and IPv4 Sunset

=== 7.1. IPv6-Only Mode (RECOMMENDED for Localhost)

For localhost Groove (the default scope), IPv6-only binding is RECOMMENDED.
Every operating system capable of running Groove has a functional IPv6
loopback (`::1`). There is no legitimate reason to use IPv4 loopback for
local Groove discovery.

**Startup procedure:**

. Probe `::1` — attempt to bind a test socket to `[::1]:0`
. If successful → bind to `[::1]:PORT` only. Do not create an IPv4 socket.
. If unsuccessful (no IPv6 stack) → fall back to `127.0.0.1:PORT` and emit
  a deprecation warning to the system log.
. Report the binding mode in the Groove Manifest:

[source,a2ml]
----
@groove-manifest(version="0.1.0"):
  @system(id="my-service", version="0.1.0")
  @transport:
    @binding(protocol="ipv6", address="::1", port=6473)
    @ipv4-sunset(date="2027-01-01", status="deprecated")
  @end
@end
----

=== 7.2. IPv4 Sunset Headers

During a transition period, a Groove service that still accepts IPv4
connections MUST include sunset headers signalling that IPv4 support will
be removed:

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/groove+a2ml
Groove-IPv6-Preferred: [::1]:6473
Groove-IPv4-Sunset: 2027-01-01
Groove-Upgrade-To: ipv6
Sunset: Sat, 01 Jan 2027 00:00:00 GMT
----

[cols="2,3"]
|===
| Header | Purpose

| `Groove-IPv6-Preferred`
| The IPv6 endpoint the client SHOULD use for all subsequent traffic

| `Groove-IPv4-Sunset`
| The date after which IPv4 will no longer be accepted (ISO 8601)

| `Groove-Upgrade-To`
| Directive to the client to upgrade its connection to IPv6

| `Sunset` (RFC 8594)
| Standard HTTP sunset header for the IPv4 endpoint
|===

This follows the same pattern as HTTP Strict Transport Security (HSTS):
the client is told to upgrade, and all subsequent connections use the
preferred transport.

=== 7.3. Strip-and-Upgrade on Arrival

When a Groove service receives an IPv4 connection and IPv6 is available,
the service SHOULD perform a strip-and-upgrade:

. Accept the IPv4 connection (refusing causes a confusing connection error)
. Read the Groove request
. Respond with the manifest including `Groove-Upgrade-To: ipv6` header
. For connection handshake (`POST /.well-known/groove/connect`):
  accept the connection but return the IPv6 endpoint for all data-plane
  traffic
. All subsequent data-plane traffic uses IPv6 exclusively
. The IPv4 socket serves only as a discovery doorbell — the real
  conversation happens on IPv6

[source]
----
Client (IPv4)                          Groove Service
     │                                       │
     │──── GET /.well-known/groove ────────→ │  ← IPv4 accepted (doorbell)
     │                                       │
     │←── 200 OK + Groove-Upgrade-To: ipv6 ──│  ← "Come to the IPv6 entrance"
     │                                       │
     │──── POST /connect (on [::1]:6473) ──→ │  ← IPv6 connection (real door)
     │                                       │
     │←──────── Connected (IPv6) ────────────│  ← All subsequent traffic is IPv6
----

=== 7.4. IPv6 Strict Mode

Groove services MAY support an IPv6 strict mode where IPv4 connections are
refused entirely:

[source,a2ml]
----
@transport:
  @binding(protocol="ipv6-strict", address="::1", port=6473)
  @ipv4(status="disabled")
@end
----

In strict mode:

. No IPv4 socket is created
. IPv4 probes receive no response (connection refused)
. The manifest declares `ipv4.status = "disabled"`

This is RECOMMENDED for new deployments. Legacy compatibility mode
(dual-stack or strip-and-upgrade) is available for transition.

=== 7.5. Frame-Level Proxy (RECOMMENDED for Native Services)

For native Groove services (Zig, Elixir, Rust), the RECOMMENDED
strip-and-upgrade strategy operates at the TCP stream frame level
(Layer 4), not the HTTP level (Layer 7). This provides transparent,
zero-copy IPv4→IPv6 translation with no client reconnection.

**Architecture:**

[source]
----
Client (IPv4)              Frame Proxy (Zig)           Groove Service
     │                          │                           │
     │── TCP connect ─────────→ │                           │
     │                          │── TCP connect [::1] ────→ │
     │                          │                           │
     │── bytes ───────────────→ │── splice(2) ────────────→ │  ← zero-copy
     │                          │                           │
     │←── bytes ──────────────── │←── splice(2) ────────────│  ← zero-copy
     │                          │                           │
     │   (client never knows    │   (service only sees     │
     │    it's being proxied)   │    IPv6 connections)     │
----

The proxy accepts IPv4 TCP connections and opens a corresponding IPv6
TCP connection to the real Groove service on `[::1]:PORT`. Bytes are
spliced bidirectionally using the kernel's `splice(2)` syscall on Linux,
which transfers data between file descriptors without copying to
userspace.

**Properties:**

. **Zero-copy:** On Linux, `splice(2)` moves data between sockets via
  kernel pipe buffers. No bytes are copied to userspace. On macOS and
  Windows, the proxy falls back to a small userspace buffer (4KB).
. **Transparent:** The client does not know it is being proxied. No
  HTTP upgrade headers, no reconnection, no protocol awareness needed.
. **Protocol-agnostic:** Works for any protocol inside the TCP stream:
  HTTP, WebSocket, gRPC, binary Groove frames, or raw bytes. The proxy
  does not parse the content.
. **No elevated privileges:** Runs in userspace. No kernel modules,
  nftables rules, or eBPF programs required.
. **Controllable:** The proxy can log connections, apply rate limiting,
  reject based on source address, and collect metrics — all before
  splicing begins.

**Zig reference implementation:**

[source,zig]
----
const std = @import("std");

/// Groove IPv4→IPv6 frame-level proxy.
/// Accepts IPv4 connections, splices them to the IPv6 groove service.
pub fn startProxy(
    ipv4_port: u16,
    ipv6_port: u16,
    allocator: std.mem.Allocator,
) !void {
    // Bind IPv4 doorbell socket
    const ipv4_addr = std.net.Address.parseIp4("127.0.0.1", ipv4_port);
    var ipv4_server = try std.net.StreamServer.init(.{
        .reuse_address = true,
    });
    defer ipv4_server.deinit();
    try ipv4_server.listen(ipv4_addr);

    while (true) {
        const ipv4_conn = try ipv4_server.accept();

        // Spawn a thread to handle the proxied connection
        _ = try std.Thread.spawn(.{}, handleProxy, .{
            ipv4_conn.stream,
            ipv6_port,
        });
    }
}

fn handleProxy(ipv4_stream: std.net.Stream, ipv6_port: u16) void {
    defer ipv4_stream.close();

    // Connect to the real groove service on IPv6
    const ipv6_addr = std.net.Address.parseIp6("::1", ipv6_port) catch return;
    const ipv6_stream = std.net.tcpConnectToAddress(ipv6_addr) catch return;
    defer ipv6_stream.close();

    // Bidirectional splice
    const t = std.Thread.spawn(.{}, splicePair, .{
        ipv6_stream, ipv4_stream,
    }) catch return;

    splicePair(ipv4_stream, ipv6_stream);
    t.join();
}

fn splicePair(from: std.net.Stream, to: std.net.Stream) void {
    var buf: [4096]u8 = undefined;
    while (true) {
        const n = from.read(&buf) catch break;
        if (n == 0) break;
        to.writeAll(buf[0..n]) catch break;
    }
}
----

NOTE: The reference implementation above uses a userspace buffer for
portability. On Linux, replace `read`/`writeAll` with `splice(2)` via
`std.os.linux.splice()` for true zero-copy operation.

**When to use frame-level vs HTTP-level:**

[cols="2,2,2"]
|===
| Context | Strategy | Why

| Native Zig/Elixir/Rust service
| Frame-level proxy
| Zero-copy, transparent, no reconnection

| Browser extension
| HTTP-level upgrade headers
| Extensions use `fetch()`, cannot do raw TCP

| Legacy bridge
| Frame-level proxy
| Legacy system doesn't understand upgrade headers

| LAN discovery
| Frame-level proxy
| Avoids exposing IPv4 upgrade semantics across the network
|===

=== 7.6. Client Behaviour

Groove clients (including browser extension harnesses) MUST:

. Probe `[::1]` first for all local discovery
. Respect `Groove-Upgrade-To: ipv6` headers by reconnecting on IPv6
. Cache the IPv6 preference for the duration of the session
. NOT fall back to IPv4 after a successful IPv6 connection
. Fall back to IPv4 ONLY if IPv6 probing produces no results AND the
  `Groove-IPv4-Sunset` header is absent or the sunset date has not passed

=== 7.6. LAN IPv6 Forward Strategy

For LAN discovery, IPv6 provides inherent advantages:

. **Link-local addresses** (`fe80::...`) exist on every interface without
  DHCP — zero-configuration addressing
. **mDNS multicast** over `ff02::fb` is cleaner than IPv4 `224.0.0.251`
. **No NAT** — IPv6 link-local addresses are directly reachable on the
  segment without NAT traversal

Groove services offering LAN discovery SHOULD use IPv6 link-local multicast
exclusively. IPv4 LAN discovery (mDNS over `224.0.0.251`) is OPTIONAL and
SHOULD be sunset on the same timeline as IPv4 loopback.

== 8. Summary

[cols="2,3"]
|===
| Requirement | Specification

| Localhost binding | IPv6-only `[::1]:PORT` RECOMMENDED. Dual-stack as transition.
| Probe order | `[::1]` first, `127.0.0.1` fallback only if IPv6 absent
| IPv4 sunset | `Groove-IPv4-Sunset` header with date, `Groove-Upgrade-To: ipv6`
| Strip-and-upgrade | Accept IPv4 as doorbell, redirect data plane to IPv6
| Strict mode | IPv6-strict available: no IPv4 socket created at all
| CORS | `Access-Control-Allow-Origin: *` on all groove endpoints
| Preflight | `OPTIONS` → `204 No Content` with CORS headers
| LAN discovery | mDNS/DNS-SD over IPv6 multicast `ff02::fb`, opt-in only
| Default scope | Localhost only (loopback interface)
| Data plane | Any transport (HTTP/WS/gRPC/WebRTC/Unix/BEAM), declared in ABI
| Client behaviour | Cache IPv6 preference, never fall back after successful IPv6
|===
